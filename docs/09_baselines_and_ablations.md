# Baselines, ablations и сравнение с MLP

Документ задаёт **методологию** сравнения. Численные результаты после каждого прогона лежат в:

- `artifacts/metrics/benchmark_results.json`
- `artifacts/reports/benchmark_report.md` (таблица + краткие примечания)

Пересчёт:

```bash
python scripts/compare_baselines.py
```

Опции: `--ablations full wo_context`, `--no-xgb`, `--mlp-per-ablation` (долго), `--skip-production-mlp-row`.

## 1. Зачем baselines

| Модель | Роль в эксперименте |
|--------|---------------------|
| **Logistic regression** | Линейный потолок в пространстве преобразованных признаков; интерпретируемый коэффициентный уровень. |
| **Random Forest** | Нелинейности и взаимодействия без явного one-hot «взрыва» размерности на стороне модели (деревья на смешанных признаках после того же препроцессинга, что и у MLP). |
| **XGBoost** | Сильный градиентный бустинг по таблицам; часто SOTA среди классиков на структурированных данных. |

Все три обучаются на **той же матрице**, что получается после `ColumnTransformer`, fit **только на train**, оценка на **том же test**, что и у нейросети. Порог бинаризации совпадает с `classification_threshold` в конфиге.

## 2. Нейросеть «лучше» или «хуже»

Ожидаемо (и типично для демо-таблиц умеренного размера):

- **XGBoost / RF** часто конкурируют с MLP или превосходят его по ROC-AUC / F1 на табличных данных без огромного объёма обучения.
- **MLP** может выигрывать при достаточном объёме данных, тонкой настройке и если нелинейная граница хорошо аппроксимируется; плюс единый PyTorch-стек для прод-инференса и расширений (embeddings, мультимодальность).

**Вывод для репозитория:** нейросеть здесь — **целевой артефакт** по ТЗ (диплом, единый пайплайн torch → API), а baselines — **контроль качества**: если MLP сильно проигрывает RF/XGB при том же сплите, имеет смысл увеличить данные, подобрать архитектуру/регуляризацию или оставить бустинг как прод-модель, а MLP как исследовательский baseline.

Строка **`mlp_production`** в отчёте подтягивается из `artifacts/metrics/test_metrics.json` после полного `scripts/train.py` (полный набор признаков, полное число эпох). Она **не** гарантированно сопоставима по числу эпох с `mlp_short` из `--mlp-per-ablation`.

## 3. Ablation-конфигурации

Определены в `app/features/ablation.py`:

| Имя | Что убрано (инженерная гипотеза) |
|-----|----------------------------------|
| `full` | Ничего; референс. |
| `wo_context` | Ночь, выходной, погода, трафик, зона риска — «контекст среды». |
| `wo_user_history` | Задержки платежей, инциденты, рейтинг. |
| `wo_trip_dynamics` | Длительность, дистанция, скорости, флаг просрочки завершения. |

Для каждого ablation препроцессор и модели обучаются заново на train-подмножестве колонок. Ожидаемо метрики **падают**, если удалённая группа несёт сигнал (см. JSON отчёт: какой ablation бьёт сильнее по F1/ROC-AUC).

## 4. Как читать таблицу

- Сравнивайте **одну колонку** (например `f1` или `roc_auc`) внутри одного `ablation` между моделями.
- Сравнение **между ablation** для одной модели показывает важность групп признаков.
- При дисбалансе классов ориентир — **F1 + ROC-AUC**, не только accuracy.

## 5. Пример интерпретации (демо)

На синтетической выборке нередко **логистическая регрессия** даёт высокий ROC-AUC за счёт почти линейной разделимости в пространстве признаков после генерации таргета, в то время как **MLP** без тюнинга может оказаться ближе к RF/XGB по F1 в зависимости от порога и архитектуры. **Random Forest** может поднять accuracy за счёт консервативных предсказаний класса 1 (выше precision, ниже recall) — это видно по сравнению precision/recall в таблице. Такой паттерн **нормален** для демо: он показывает, что «нейросеть в репозитории» — не автоматически лучший скор на таблице, и требует сопоставления с baselines. Актуальные цифры всегда смотрите в `artifacts/reports/benchmark_report.md`.
