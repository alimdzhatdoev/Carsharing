## 1. Формальная постановка задачи

### 1.1. Исходные объекты и цель

Рассматривается множество **объектов наблюдения** — завершённых поездок в системе краткосрочной аренды транспорта (каршеринга). Каждой поездке сопоставляется вектор признаков, формируемый из данных о пользователе, параметрах поездки, характеристиках автомобиля и контексте (время суток, погода, дорожная обстановка и т.д.).

**Целевая переменная** — бинарная метка \(y \in \mathcal{Y} = \{0, 1\}\):

- \(y = 0\) — поездка относится к **нормальному** (типичному) использованию сервиса;
- \(y = 1\) — поездка относится к классу **повышенного риска** (потенциально проблемная в смысле инцидента, нарушения правил, мошенничества или значимого негативного эффекта для оператора).

В терминах машинного обучения ставится задача **обучения с учителем**: по обучающей выборке пар «признаки — метка» построить алгоритм, обобщающий закономерности на новые поездки.

### 1.2. Пространство признаков и отображение

Пусть «сырые» табличные признаки описываются вектором \(\tilde{\mathbf{x}}\) фиксированной размерности (набор колонок согласован с описанием данных в проекте). Для числовой обработки вводится **детерминированное преобразование** \(\Phi\) (импутация пропусков, стандартизация числовых признаков, кодирование категориальных признаков), задаваемое конвейером предобработки и **обучаемое только на обучающей подвыборке**, чтобы исключить утечку информации с контрольных выборок:

\[
\mathbf{x} = \Phi(\tilde{\mathbf{x}}) \in \mathcal{X} \subseteq \mathbb{R}^d .
\]

**Классификатор** в проекте реализован как композиция:

1. отображение \(\Phi\) (фиксируется после обучения и сохраняется как артефакт);
2. нейросетевая функция \(h_\theta: \mathbb{R}^d \to \mathbb{R}\), выдающая **логит** \(z = h_\theta(\mathbf{x})\);
3. оценка вероятности положительного класса \(\hat{p}(\mathbf{x}) = \sigma(z) = 1/(1+e^{-z})\);
4. пороговое решение \(\hat{y} = \mathbb{I}[\hat{p}(\mathbf{x}) \ge \tau]\), где \(\tau \in (0,1)\) — порог, задаваемый конфигурацией системы.

### 1.3. Обучающая выборка и критерий обучения

Обучающая выборка \(\mathcal{D} = \{(\mathbf{x}_i, y_i)\}_{i=1}^N\) предполагается полученной из одного и того же **генерального механизма** формирования поездок (в демонстрационной реализации — синтетическая генерация с доменно мотивированной логикой метки; в прикладной постановке — выгрузка из информационных систем оператора с последующей разметкой или правилами формирования \(y\)).

Параметры \(\theta\) нейросети подбираются минимизацией **эмпирического риска**: средней величины функции потерь на обучающей выборке (в реализации — `BCEWithLogitsLoss` с учётом дисбаланса классов через `pos_weight`; подробности — `docs/04_modeling_decisions.md`). Контроль переобучения — валидационная выборка, ранняя остановка по валидационному критерию, регуляризация через dropout и весовой коэффициент в оптимизаторе.

### 1.4. Требования к решению

Искомое решение должно:

1. допускать **воспроизводимое** переобучение на новых данных при сохранении протокола сплита и предобработки;
2. сохранять артефакты (веса модели, параметры \(\Phi\), конфигурацию) для **инференса** в прикладном сервисе (REST API);
3. допускать **сравнение** с базовыми моделями на тех же выборках (`docs/09_baselines_and_ablations.md`).

## 2. Предметная область: каршеринг

### 2.1. Сущность сервиса

**Каршеринг** (краткосрочная аренда автомобиля по модели «доступ по запросу») предполагает, что пользователь через мобильное приложение получает доступ к автомобилю из распределённого парка, оплачивает использование по факту (время, пробег, тариф), а оператор обеспечивает обслуживание, страхование, технику безопасности и соблюдение правил платформы.

### 2.2. Участники и данные

В типовой экосистеме выделяются: **пользователь** (профиль, история поездок, платежи), **поездка** (телематика, время, маршрутные характеристики в агрегированном виде), **автомобиль** (класс, возраст, пробег, уровень топлива/заряда), **внешний контекст** (погода, трафик, характеристики зоны). Эти сущности естественно отображаются в **табличный признаковый вектор** на уровне одной поездки.

### 2.3. Практическая мотивация задачи классификации риска

Оператор каршеринга сталкивается с редкими, но затратными событиями: повреждения, споры по оплате, злоупотребления, конфликты с третьими лицами. Полный ручной разбор всех поездок невозможен; требуется **приоритизация** — выделение подмножества поездок для усиленного контроля, связи с поддержкой или внутреннего аудита. Модель бинарной классификации выступает **скоринговым инструментом**: она не заменяет регламент и человеческое решение, но снижает поисковое пространство для операционных служб.

### 2.4. Соответствие признаков домену

Набор признаков в репозитории согласован с перечисленными сущностями (см. `docs/03_data_description.md`): возраст и история пользователя, параметры движения и длительности поездки, характеристики ТС, индикаторы ночи/выходных, погодные и дорожные условия, категория зоны. Такое описание позволяет в тексте ВКР связать **формальную постановку** с **предметной интерпретацией** признаков.

## 3. Метрики качества бинарной классификации

### 3.1. Матрица ошибок

Для порогового классификатора на контрольной выборке фиксируются:

- **TP** (true positive) — верно предсказан класс 1;
- **TN** (true negative) — верно предсказан класс 0;
- **FP** (false positive) — ошибочно предсказан класс 1 (ложная тревога);
- **FN** (false negative) — ошибочно предсказан класс 0 (пропуск проблемной поездки).

### 3.2. Основные метрики

**Доля верных ответов (accuracy):**

\[
\mathrm{Accuracy} = \frac{\mathrm{TP} + \mathrm{TN}}{\mathrm{TP} + \mathrm{TN} + \mathrm{FP} + \mathrm{FN}}.
\]

При **дисбалансе классов** accuracy может быть обманчиво высокой, если доминирует класс 0.

**Точность (precision)** для класса 1:

\[
\mathrm{Precision} = \frac{\mathrm{TP}}{\mathrm{TP} + \mathrm{FP}}.
\]

Характеризует долю истинных проблемных поездок среди всех отмеченных моделью как проблемные.

**Полнота (recall)** для класса 1:

\[
\mathrm{Recall} = \frac{\mathrm{TP}}{\mathrm{TP} + \mathrm{FN}}.
\]

Характеризует долю обнаруженных реальных случаев класса 1.

**F1-мера** — гармоническое среднее precision и recall:

\[
F_1 = \frac{2 \cdot \mathrm{Precision} \cdot \mathrm{Recall}}{\mathrm{Precision} + \mathrm{Recall}}.
\]

Удобна как **скалярный компромисс** при несбалансированных классах.

### 3.3. ROC-AUC

Варьируя порог \(\tau\), строится зависимость **TPR** (true positive rate) от **FPR** (false positive rate) — ROC-кривая. **AUC** (площадь под ROC-кривой) оценивает качество **ранжирования** объектов по оценке \(\hat{p}(\mathbf{x})\) без привязки к одному фиксированному порогу. Для бинарной задачи с обоими классами на контрольной выборке AUC принимает значения в \([0,1]\); значение 0.5 соответствует случайному угадыванию.

### 3.4. Интерпретация в контексте каршеринга

- Высокий **recall** при умеренной **precision** — меньше пропущенных рисковых поездок, но больше ложных тревог и нагрузки на поддержку.
- Высокая **precision** при низком **recall** — реже беспокоят «хороших» пользователей, но выше риск пропустить инцидент.

Выбор порога \(\tau\) и весов в функции потерь должен согласовываться с **бизнес-ограничениями** (см. `docs/02_business_context.md`, `docs/05_metrics_and_validation.md`).

## 4. Обоснование выбора нейросетевой модели

### 4.1. Тип входных данных и класс моделей

Признаки носят **смешанный** характер (числовые и категориальные поля после кодирования образуют вещественный вектор фиксированной размерности). Такой формат естественно обрабатывается **полносвязной нейросетью (MLP)**: последовательность аффинных преобразований и нелинейностей аппроксимирует границу решений в \(\mathbb{R}^d\).

### 4.2. Почему нейросеть целесообразна в рамках работы

1. **Нелинейность и взаимодействия.** Риск проблемной поездки в предметной области может зависеть от **сочетаний** факторов (например, ночное время, высокая скорость, неблагоприятный контекст). MLP в принципе способна аппроксимировать такие зависимости без явного конструирования полиномиальных признаков.

2. **Единый инженерный стек.** Использование **PyTorch** обеспечивает прозрачный цикл: обучение → сохранение весов → загрузка в сервис инференса (FastAPI), что соответствует требованиям к **прикладному** характеру работы.

3. **Сопоставимость с базовыми методами.** В репозитории реализовано сравнение с логистической регрессией, случайным лесом и градиентным бустингом на **идентичных** сплитах и после **одинаковой** предобработки (`docs/09_baselines_and_ablations.md`). Это позволяет в ВКР аргументированно заявить: нейросеть рассматривается не как «единственно возможная» модель, а как **основной объект исследования** при наличии контрольных baseline.

4. **Когда нейросеть может не быть лучшей.** На табличных данных часто сильны **Random Forest** и **XGBoost**; логистическая регрессия может быть конкурентоспособна при почти линейной разделимости после \(\Phi\). Фактическое соотношение метрик фиксируется в `artifacts/reports/benchmark_report.md` после запуска `scripts/compare_baselines.py`. В тексте диплома это следует трактовать как **эмпирический результат на используемом датасете**, а не как универсальный закон.

### 4.3. Связь с целью ВКР

Выбор MLP согласуется с формулировкой темы о **разработке нейросетевой модели классификации** для задач каршеринга: акцент делается на полном пайплайне (данные — модель — оценка — внедрение в виде API), а baselines выполняют роль **научной корректности** сравнения.

## 5. Ограничения исследования

1. **Данные.** В демонстрационной конфигурации используется **синтетический** датасет с правилом формирования метки, имитирующим доменные факторы. Обобщение выводов на конкретного оператора без валидации на **реальных** логах и согласованной разметке **не гарантируется**.

2. **Разметка целевой переменной.** Операционное определение класса \(y=1\) в бою требует юридически и продуктово согласованных критериев (инциденты, chargeback, решения модерации). В работе зафиксирована **упрощённая** постановка.

3. **Каузальность.** Модель устанавливает **ассоциации** между признаками и меткой, но не доказывает причинность поведения пользователя или причинность риска без отдельного каузального анализа.

4. **Справедливость и дискриминация.** Признаки (возраст, география и т.д.) могут вносить **смещение**; автоматическое использование скора для санкций к пользователям требует отдельного правового и этического аудита.

5. **Масштаб и продакшен.** Отсутствуют нагрузочное тестирование API на уровне промышленного трафика, онлайн-мониторинг дрейфа данных и полноценная система версионирования моделей уровня MLOps — в репозитории заложены только **базовые** элементы.

6. **Интерпретируемость.** Архитектура MLP не даёт простой линейной интерпретации весов в исходных признаках; пост-hoc методы (SHAP и др.) в объём работы не входят.

## 6. Направления дальнейшей работы

1. Замена синтетических данных **выгрузками** из промышленных систем и построение разметки по согласованным бизнес-правилам.
2. Расширение семейства моделей: **калибровка** вероятностей, ансамбли MLP + бустинг, настройка порога \(\tau\) по cost-sensitive критериям.
3. **Мониторинг дрейфа** признаков и периодическое переобучение; интеграция с feature store / DVC / MLflow.
4. Исследование **многоклассовой** постановки (тип инцидента / уровень риска) при достаточном объёме размеченных данных.
5. Подключение **неструктурированных** данных (текст обращений) в мультимодельную архитектуру.
6. Расширение блока **объяснимости** (SHAP, LIME) для согласования с требованиями к прозрачности решений.
7. Юридическая и организационная **политика использования** скора (человек в контуре, ограничения автоматических санкций).
