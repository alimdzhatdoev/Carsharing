# Постановка ML-задачи

Документ фиксирует **формальную постановку**, границы решения и инженерные критерии готовности демо-репозитория. Обновляется при смене таргета или контракта данных.

Для **дипломной работы** (академическое изложение с обозначениями \(\mathcal{X}, \mathcal{Y}\), преобразованием \(\Phi\), связью с предметной областью и метриками) используйте **`docs/DIPLOMA_THESIS_BASE.md`**; текущий файл остаётся кратким инженерным конспектом.

## 1. Возможные сценарии классификации в каршеринге

| # | Сценарий | Supervised | Табличный срез | Комментарий |
|---|----------|------------|----------------|-------------|
| 1 | Вероятность мошеннической поездки | Да | Да | Редкий класс, тяжёлая разметка |
| 2 | Риск опасного вождения / инцидента | Да | Да | Телеметрия + контекст |
| 3 | Просрочка завершения аренды | Да | Да | Операционная задача |
| 4 | Сегмент поведения пользователя | Да / weak | Да | Часто multi-class |
| 5 | Класс обращения в поддержку | Да | Таблица + текст | NLP вне текущего scope |

## 2. Выбранный сценарий (зафиксировано)

**Бинарная классификация: риск проблемной поездки.**

| Класс | Смысл (операционное определение для демо) |
|-------|---------------------------------------------|
| `0` | Поездка вписывается в ожидаемое использование; эскалация не требуется. |
| `1` | Повышенный риск инцидента, нарушения правил, мошенничества или сильного негатива для сервиса; кандидат на приоритетный разбор. |

**Тип задачи:** supervised learning, **бинарная классификация**, признаки — **таблица** «одна строка = одна завершённая поездка» (агрегаты на момент закрытия сессии).

## 3. Формальные определения

- **Единица наблюдения:** поездка `trip_id` (идентификатор не используется моделью).
- **Пространство входа:** вектор признаков после единого препроцессинга фиксированной размерности (числа + one-hot категорий), см. `docs/03_data_description.md` и `docs/08_data_pipeline.md`.
- **Выход модели:** логит бинарного класса; вероятность положительного класса — `sigmoid(logit)`.
- **Решение по классу:** `probability >= τ`, где `τ` — `classification_threshold` в конфиге (по умолчанию 0.5).

## 4. Обоснование выбора сценария

- Реалистичен для каршеринга (поддержка, модерация, антифрод как **один из сигналов**).
- Хорошо ложится на табличный датасет без NLP.
- Бинарный таргет проще согласовать с бизнес-действием «эскалировать / не эскалировать», чем 4-классовая схема без жёстких определений границ классов.
- MLP на закодированных признаках — воспроизводимый baseline с единым стеком train → артефакты → API.

**Резерв на развитие:** многокласс (normal / risky / fraudulent / incident_related) после утверждения определений классов и объёма разметки.

## 5. Вне scope (явно)

- Персональные рекомендации маршрутов, динамическое ценообразование.
- Обработка неструктурированных обращений (текст чата) в текущей версии кода.
- Причинно-следственные выводы («модель доказала вину») — только **скоринг**.

## 6. Критерии готовности демо-репозитория

- Контракт сырого CSV проверяется скриптом (`scripts/prepare_data.py`).
- Train/val/test с стратификацией; препроцессинг **fit только на train**; артефакты сохраняются.
- Есть отчёты метрик и `model_report.md` после `evaluate`.
- API поднимается на обученных артефактах; Docker описан в `docs/07_deployment.md`.

## 7. Бизнес-ценность и применение

- Приоритизация очередей модерации и поддержки.
- Входной сигнал в rule engine (совместно с лимитами, чёрными списками, KYC).
- Офлайн: переобучение, анализ ошибок, подбор порога `τ`.

## 8. Допущения демо

- Таргет и часть распределений **синтетические**; логика генерации имитирует доменные факторы риска (см. `scripts/generate_demo_data.py`).
- Прод-разметка должна быть определена отдельно (инциденты, chargeback, решения модерации).
