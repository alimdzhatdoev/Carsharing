# Метрики и валидация

Формальные определения метрик (через TP/TN/FP/FN, ROC-AUC) и их роль в тексте диплома — **`docs/DIPLOMA_THESIS_BASE.md`**, §3. Ниже — инженерная конкретика проекта.

## Используемые метрики

- **Accuracy** — доля верных ответов; удобна при сбалансированных классах.
- **Precision** — доля истинных «проблемных» среди предсказанных как 1; важна, чтобы не перегружать модерацию ложными тревогами.
- **Recall** — доля найденных реальных проблемных поездок; важна, чтобы не пропускать риск.
- **F1-score** — гармоническое среднее precision и recall; баланс при дисбалансе.
- **ROC-AUC** — качество ранжирования по вероятности независимо от фиксированного порога (для бинарной задачи).

## Почему accuracy недостаточна

При дисбалансе (например, 85% класса 0) модель, предсказывающая всегда 0, даёт высокую accuracy, но **бесполезна** для поиска проблемных поездок. Нужны **precision, recall, F1, ROC-AUC**.

## Валидация

- Разбиение **train / validation / test** с фиксированным `random_seed`.
- Препроцессинг: **fit только на train**; val/test/inference — только `transform` с сохранёнными артефактами (нет утечки).
- На **validation** — early stopping и выбор лучшего чекпоинта.
- Финальная оценка на **test** (и отдельный скрипт `evaluate` для воспроизводимости).

## Дисбаланс классов

- В демо используются **веса классов** в `BCEWithLogitsLoss` (`pos_weight`), вычисляемые по частотам в обучающей выборке (формула и обоснование — `docs/04_modeling_decisions.md`).
- Альтернативы (задокументированы): `WeightedRandomSampler`, focal loss — могут быть добавлены при сильном дисбалансе.

## Порог классификации

- По умолчанию 0.5 на вероятности после sigmoid; параметр `classification_threshold` в конфиге позволяет сдвигать компромисс FP/FN под бизнес.

## Интерпретация в каршеринге

- Высокий **recall**, низкий **precision** — больше ложных тревог, очередь модерации перегружена.
- Высокий **precision**, низкий **recall** — много реальных проблем пропускается (опасно для репутации и убытков).
- **False positive** — лишняя работа поддержки, раздражение честных пользователей.
- **False negative** — пропущенный инцидент/мошенничество.
