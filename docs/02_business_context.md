# Бизнес-контекст

Связь ML-модели с процессами каршеринга: кто потребитель скора, какие решения допустимы, какие риски учитывать. Документ согласуется с `01_problem_statement.md` и `05_metrics_and_validation.md`.

Раздел **«Предметная область: каршеринг»** в академическом изложении для ВКР — в `docs/DIPLOMA_THESIS_BASE.md` (§2); здесь — прикладной и организационный акцент.

## 1. Заинтересованные стороны

| Роль | Интерес к модели |
|------|-------------------|
| Операции / модерация | Сократить объём ручного просмотра, не теряя критичные кейсы |
| Поддержка (L2/L3) | Приоритизация тикетов, связанных с «тяжёлыми» поездками |
| Антифрод / Risk | Дополнительный числовой сигнал к правилам и графам |
| Продукт / аналитика | Понимание факторов и качества скоринга (офлайн) |
| Юридический / Compliance | Соответствие автоматизированным решениям и персональным данным |

## 2. Где используется предсказание

1. **Пост-фактум после поездки** (текущий демо-срез): агрегированные признаки известны, таргет в обучении синтетический.
2. **Операционный дашборд:** сортировка и фильтр по `probability_positive`.
3. **Триггеры в rule engine:** порог `τ` и комбинация с жёсткими правилами (например, множественные инциденты).
4. **Пост-аналитика:** срезы по зонам, тарифам, времени суток для улучшения продукта.

Интеграция в реальном бэкенде: **REST** (см. `docs/06_api_usage.md`) или внутренняя очередь событий; демо отдаёт только HTTP.

## 3. Решения на основе результата

Модель не назначает санкции сама по себе — рекомендуется **политика принятия решений** (пример):

| Уровень риска (по порогам) | Пример действий |
|----------------------------|-----------------|
| Низкий (`p < τ_low`) | Обычный поток |
| Средний (`τ_low ≤ p < τ_high`) | Логирование, мягкие уведомления, выборочная выборка для аудита |
| Высокий (`p ≥ τ_high`) | Задача модератору, доп. проверка платежа, блок повторной аренды до разбирательства |

В репозитории задан **один** порог `classification_threshold` (`configs/config.yaml`); многоуровневая политика — расширение конфига и API.

## 4. Метрики и бизнес-последствия ошибок

- **False positive:** лишняя нагрузка на людей, негатив у добросовестных пользователей.
- **False negative:** пропуск инцидента, мошенничества, ущерб и репутационные потери.

Выбор `τ` — компромисс; в демо фиксируется в конфиге, на проде — по cost matrix и ограничениям регулятора.

## 5. Ограничения и риски

- **Смещение:** возраст, география, тариф могут коррелировать с защищёнными признаками; нужны политики fair lending / non-discrimination по юрисдикции.
- **Дрейф:** новые города, сезон, смена парка ТС — мониторинг распределений входов и периодическое переобучение.
- **Интерпретируемость:** MLP — не линейная модель; для объяснимости пользователю нужны отдельные инструменты (SHAP и т.д.) — вне минимального scope.
- **Разметка:** в демо таргет задан генератором; в бизнесе определение `target_class=1` должно быть задокументировано (SLA, типы инцидентов).

## 6. Соответствие и этика

- Автоматические блокировки только по скору без человека могут быть **запрещены или ограничены**; фиксируется как организационное решение, не как поведение кода в репозитории.
- Логи предсказаний на проде — с политикой хранения ПДн.
