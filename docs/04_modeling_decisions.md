# Решения по моделированию

Аргументация выбора нейросетевой модели в логике ВКР (цели работы, сравнение с baselines, оговорки) — **`docs/DIPLOMA_THESIS_BASE.md`**, §4. Ниже — технические детали реализации.

## Почему нейросетевая модель

- **MLP (многослойный перцептрон)** на закодированных табличных признаках — универсальный нелинейный baseline, единый стек PyTorch: обучение → чекпоинт → инференс в API.
- Возможное расширение: **embedding** категорий, мультимодальные входы без смены фреймворка.

## Функция потерь (детально)

### Формулировка задачи на уровне выхода сети

Используется **один выходной логит** \(z \in \mathbb{R}\) на объект. Вероятность «проблемной» поездки (класс 1) задаётся сигмоидой \(\sigma(z) = 1/(1+e^{-z})\).

**Почему один логит, а не два (softmax)?** Для бинарной задачи эквивалентны: softmax на двух логитах избыточен; один логит + `BCEWithLogitsLoss` — стандартная связка в PyTorch, меньше параметров на выходном слое.

### BCEWithLogitsLoss

PyTorch `nn.BCEWithLogitsLoss` объединяет **сигмоиду и бинарную кросс-энтропию** в одной численно устойчивой форме (log-sum-exp), вместо `sigmoid` → `BCELoss` по отдельности (меньше потери точности и градиентов при экстремальных \(z\)).

Для батча с метками \(y_i \in \{0,1\}\) и логитами \(z_i\) без весов классов:

\[
\mathcal{L} = -\frac{1}{N}\sum_i \Big( y_i \log \sigma(z_i) + (1-y_i)\log(1-\sigma(z_i)) \Big).
\]

### Учёт дисбаланса: `pos_weight`

В проекте используется **`pos_weight`** (скаляр `torch.Tensor` длины 1), задаваемый как отношение числа отрицательных примеров к положительным на **train**:

\[
\text{pos\_weight} = \frac{N_{y=0}}{N_{y=1}}.
\]

Это эквивалентно увеличению вклада ошибок на положительном классе в лоссе и близко по духу к **взвешенной** бинарной кросс-энтропии без изменения самих меток. Альтернативы (не используются в коде по умолчанию, но допустимы): `WeightedRandomSampler`, focal loss — фиксируются как возможное развитие в `05_metrics_and_validation.md`.

### Связь с порогом при инференсе

Лосс оптимизирует **калибровку логитов** в среднем по выборке; порог \(\tau\) на \(\sigma(z)\) для бизнес-решения задаётся отдельно (`classification_threshold`). Подбор \(\tau\) по валидации — нормальная практика; **по test** только финальная отчётность.

## Сравнение с базовыми моделями

Эмпирическое сравнение на тех же сплитах и том же препроцессинге: `scripts/compare_baselines.py`, методология — `docs/09_baselines_and_ablations.md`.

| Подход | Плюсы | Типичное положение относительно MLP на таблицах |
|--------|--------|-----------------------------------------------|
| **Logistic regression** | Быстро, интерпретируемые веса в пространстве признаков | Часто ниже по нелинейным паттернам |
| **Random Forest** | Устойчивость, взаимодействия без явной архитектуры | Сильный конкурент / часто сильнее простого MLP |
| **XGBoost** | Высокое качество на многих табличных задачах | Часто сильный baseline; может превосходить MLP при умеренном объёме данных |

**Итог:** нейросеть в репозитории не обязана быть «лучше» RF/XGB; она задана **требованиями проекта** (PyTorch, единый деплой). Если бустинг стабильно лучше — рациональное инженерное решение на проде: бустинг как модель, MLP как альтернатива/ensemble.

## Ablation-анализ признаков

Удаление осмысленных групп колонок (контекст, история пользователя, динамика поездки) и повторное обучение baselines на том же протоколе — см. `app/features/ablation.py` и отчёт `artifacts/reports/benchmark_report.md`. Падение метрик указывает на значимость группы для данного датасета.

## Архитектура MLP (конфигурируемая)

- Вход: вектор после `ColumnTransformer` (числа + one-hot).
- Слои, dropout, batch norm, активация — `configs/config.yaml`.

## Ограничения

- TabTransformer / NODE и т.п. не используются; качество на сложных таблицах может быть ниже, чем у тюнингованного бустинга.
- Калибровка вероятностей на проде может потребовать отдельного шага (Platt / isotonic).
